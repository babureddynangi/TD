## Dataset Generation

Datasets are generated by `scripts/generate_benchmark_data.py` with a fixed random seed (`seed=42`) for full reproducibility.

### Defect Seeding Strategy

Each record is assigned exactly one defect category (or "clean") using weighted random sampling:

| Defect Category | Weight | Approx % | Expected Outcome |
|---|---|---|---|
| clean | 60 | 60% | PASS |
| schema_missing_field | 5 | 5% | FAIL |
| schema_type_mismatch | 3 | 3% | FAIL |
| schema_date_format | 3 | 3% | FAIL |
| schema_invalid_enum | 2 | 2% | FAIL |
| dq_null_mandatory | 5 | 5% | FAIL |
| dq_duplicate_account | 3 | 3% | FAIL |
| dq_invalid_state | 3 | 3% | FAIL |
| dq_invalid_zip | 2 | 2% | FAIL |
| dq_future_date | 2 | 2% | FAIL |
| bus_available_credit | 3 | 3% | FAIL |
| bus_past_due_conflict | 3 | 3% | FAIL |
| bus_missing_close_date | 2 | 2% | FAIL |
| bus_delinquency_mismatch | 2 | 2% | FAIL |
| state_no_rule | 2 | 2% | REVIEW_REQUIRED |

Each defect is injected deterministically:

- `schema_missing_field` — removes one of: `account_id`, `tradeline_id`, `zip_code`, `reporting_date`
- `schema_type_mismatch` — sets `current_balance = "not_a_number"`
- `schema_date_format` — sets `reporting_date = "15/01/2026"` (DD/MM/YYYY, not ISO 8601)
- `schema_invalid_enum` — sets `product_type = "DEBIT_CARD"`
- `dq_null_mandatory` — sets `account_id = ""`
- `dq_duplicate_account` — reuses an `account_id` already seen in the dataset
- `dq_invalid_state` — sets `state_code` to one of: ZZ, XX, QQ, 99
- `dq_invalid_zip` — sets `zip_code` to a 3–4 digit string (too short)
- `dq_future_date` — sets `reporting_date` to 1–365 days after the fixed run date
- `bus_available_credit` — adds 50–500 to `available_credit` to break the invariant
- `bus_past_due_conflict` — sets `payment_status=CURRENT` with `past_due_amount > 0`
- `bus_missing_close_date` — sets `account_status=CLOSED` with `close_date=null`
- `bus_delinquency_mismatch` — sets `payment_status=CURRENT` with a non-CURRENT `delinquency_bucket`
- `state_no_rule` — sets `state_code` to a state not in `state_rules.json` (e.g. OR, MN, CO)

Truth labels are written to `benchmark-data/{size}_truth.json` alongside each input file.

### Fixed Run Date

All datasets use `run_date = 2026-03-01` to make future-date checks deterministic across runs.

## Benchmark Procedure

Benchmarks are run by `scripts/run_benchmark.py`.

### Pipeline Execution

The benchmark runs the full 4-stage validation pipeline locally (no AWS required):

```
Stage 1: Schema Validator   → src/handlers/schema_validate.py
Stage 2: DQ Validator       → src/handlers/dq_validate.py
Stage 3: Business Validator → src/handlers/business_validate.py
Stage 4: State Validator    → src/handlers/state_validate.py
Finalize                    → src/utils/result_builder.py
```

Each stage is called directly (not via Lambda invoke), so latency measures pure Python execution time with no network overhead. This is a local correctness and throughput benchmark, not a cloud latency measurement.

### Repetitions

Each dataset size is run 5 times. Latency statistics (mean, min, max, p95) are computed across all 5 runs. The last run's results are used for evaluation metrics.

Records are deep-copied before each run because the pipeline annotates records in place.

### Evaluation Methodology

After the pipeline runs, results are compared against truth labels:

| Metric | Definition |
|---|---|
| True Positive (TP) | Pipeline output = FAIL, truth = FAIL |
| False Positive (FP) | Pipeline output = FAIL or REVIEW_REQUIRED, truth = PASS |
| False Negative (FN) | Pipeline output = PASS, truth = FAIL |
| True Negative (TN) | Pipeline output = PASS, truth = PASS |
| Review Correct | Pipeline output = FAIL or REVIEW_REQUIRED, truth = REVIEW_REQUIRED |
| Review Missed | Pipeline output = PASS, truth = REVIEW_REQUIRED |

Precision, Recall, and F1 are computed over FAIL-class detections only (REVIEW_REQUIRED records are tracked separately as `review_correct` / `review_missed`):

```
Precision = TP / (TP + FP)
Recall    = TP / (TP + FN)
F1        = 2 × Precision × Recall / (Precision + Recall)
```

### False Positive Explanation

False positives on clean records occur when a randomly generated "clean" record happens to violate a business rule — most commonly floating-point drift in `available_credit` (e.g. `credit_limit - current_balance` differs from the stored `available_credit` by a rounding epsilon). These are genuine data quality issues in the generated data, not pipeline errors. The pipeline correctly flags them.

## Measured Results

All runs: Python 3.10, Windows, local execution, 5 repetitions, `seed=42`.

| Dataset | Records | Mean Latency | p95 Latency | Throughput | Precision | Recall | F1 |
|---|---|---|---|---|---|---|---|
| Small | 100 | 17.7ms | 33.2ms | 5,642 rec/s | 0.973 | 1.000 | 0.986 |
| Medium | 1,000 | 91.2ms | 95.1ms | 10,970 rec/s | 0.966 | 1.000 | 0.983 |
| Large | 5,000 | 489.0ms | 577.9ms | 10,225 rec/s | 0.943 | 1.000 | 0.971 |

Recall = 1.0 across all sizes — zero missed defects in any category.

Full per-category breakdown is in `large_metrics.json` → `evaluation.defect_breakdown`.

## How to Reproduce

```bash
# Install dependencies
pip install -r requirements.txt

# Generate datasets (deterministic, seed=42)
python scripts/generate_benchmark_data.py

# Run benchmark (overwrites results/*.json)
python scripts/run_benchmark.py
```

Expected output matches the tables above. Any deviation indicates an environment difference (Python version, OS, hardware).

## Scope and Limitations

| Claim | Scope |
|---|---|
| Latency figures | Local Python execution — no Lambda cold start, no network, no S3 I/O |
| Throughput figures | Single-process, single-thread — no Lambda concurrency |
| Precision/Recall/F1 | Evaluated against seeded synthetic defects — not real production data |
| State rule coverage | 10 states (CA, NY, TX, FL, IL, WA, GA, OH, PA, AZ) — 40 states → REVIEW_REQUIRED |
| ML anomaly detection | Not implemented — future work only |
| 50-state coverage | Not implemented — future work only |
| AWS deployment metrics | Not measured — CloudWatch metrics require live deployment |

These limitations are explicitly documented in the top-level README "Known Gaps and Future Work" table.
